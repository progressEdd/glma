# Plan 03-01: CLI Query Command, File Lookup, Layered Markdown Formatting

## Objective

Build the `glma query <filepath>` CLI command that looks up an indexed file in the Ladybug database and outputs compact, layered markdown with file summary, function/class signatures with docstrings, and relationship hints. This is the core query infrastructure that plans 03-02 (notebooks) and 03-03 (dependencies/metadata) extend.

## Why This First

Everything else in Phase 3 builds on this: the query command, the store query methods, and the compact formatter. Notebook compaction (03-02) is a parallel track that shares the CLI entry point. Dependency depth and metadata display (03-03) layer on top of the base query output.

Add three new methods to `LadybugStore` class in `02-worktrees/glma/src/glma/db/ladybug_store.py`:

1. `get_file_record(self, file_path: str) -> Optional[FileRecord]`:
   - Execute: `MATCH (f:File {path: $path}) RETURN f.path, f.language, f.content_hash, f.last_indexed, f.chunk_count`
   - If no rows, return None
   - Otherwise construct and return a `FileRecord` model (import FileRecord, Language from models)

2. `get_chunks_for_file(self, file_path: str) -> list[Chunk]`:
   - Move the logic from `pipeline.py`'s `_load_chunks_from_store()` function into this method
   - Execute: `MATCH (c:Chunk {file_path: $fp}) RETURN c.id, c.name, c.chunk_type, c.file_path, c.content, c.summary, c.start_line, c.end_line, c.content_hash, c.parent_id ORDER BY c.start_line`
   - Construct Chunk objects for each row: `ChunkType(row[2])` for chunk_type, `row[5] or None` for summary, `row[9] or None` for parent_id
   - Return empty list if no chunks found

3. `get_all_relationships_for_file(self, file_path: str) -> dict`:
   - Returns `dict[str, dict]` keyed by chunk_id, each value has `{"outgoing": [...], "incoming": [...]}`
   - Call `self.get_file_relationships(file_path)` for outgoing relationships
   - For each chunk in the file, call `self.get_incoming_relationships(chunk.id)` for incoming
   - For incoming relationships from other files, include `direction: "incoming"` in each dict and add the source file path by querying `MATCH (c:Chunk {id: $sid}) RETURN c.file_path`
   - Skip self-referential edges where source_id == target_id AND target_id belongs to a chunk in this file (these are unresolved targets from Phase 2)
   - Structure each relationship dict with keys: source_id, source_name, target_id, target_name, target_name_resolved, rel_type, confidence, source_line, direction (only for incoming)

After adding methods, also update `pipeline.py` to call `store.get_chunks_for_file(file_path)` instead of the inline `_load_chunks_from_store()` function. Keep `_load_chunks_from_store` as a thin wrapper that delegates to the store method for backward compatibility during this phase.

**File 2:** `02-worktrees/glma/src/glma/query/formatter.py` with the following:

Import: `from glma.models import Chunk, FileRecord, ChunkType`

Define function `format_compact_output(file_path: str, file_record: FileRecord, chunks: list[Chunk], relationships: dict, verbose: bool = False) -> str`:

The output format follows CONTEXT.md D-01 through D-04:

```
# {file_path}
*Last indexed: {file_record.last_indexed} | {file_record.chunk_count} chunks*

## Summary
{Auto-generated from chunk names and types — list top-level chunks with their types}

## Signatures
{For each top-level chunk (no parent_id), ordered by start_line:}
### {chunk.name} ({chunk_type}, L{start_line}-L{end_line})
{If chunk has attached_comments, show first comment as docstring summary}
{Relationship hint line:}
  → calls: {comma-separated list of target names from outgoing "calls" relationships}
  → called by: {comma-separated list of source names from incoming "calls" relationships}
  → imports: {comma-separated list of target names from outgoing "imports" relationships}
  → inherits: {comma-separated list of target names from outgoing "inherits" relationships}
{Only show hint lines for relationship types that exist for this chunk}

{If verbose:}
## Full Code
{For each chunk, show full content in code block with language hint}
```

Helper functions to implement:

1. `_format_summary(chunks: list[Chunk]) -> list[str]`:
   - Filter to top-level chunks (parent_id is None)
   - For each, produce: `- **{name}** ({type})` line
   - If chunk has attached_comments, append first comment stripped of quotes/markers as description

2. `_format_signature_block(chunk: Chunk, chunk_rels: dict) -> list[str]`:
   - Heading: `### {chunk.name} ({type}, L{start}-L{end})`
   - If attached_comments exists and is non-empty, show first comment as italic docstring
   - Build relationship hint lines only for types present in chunk_rels["outgoing"] and chunk_rels["incoming"]
   - Group outgoing by rel_type, format as: `  → {rel_type}: target1, target2`
   - Group incoming by rel_type, format as: `  → {rel_type} from: source1, source2`
   - For targets: use target_name if target_name_resolved is not available, otherwise use target_name_resolved
   - For self-referential (unresolved) targets, show as `? (target_name)`

3. `_format_verbose_code(chunks: list[Chunk], file_path: str) -> list[str]`:
   - Heading: `## Full Code`
   - For each chunk: heading + code block with language hint (reuse `_get_lang_hint` logic from writer.py — just check .py → "python", .c/.h → "c")

4. `_get_lang_hint(file_path: str) -> str`:
   - `.py` → "python", `.c`/`.h` → "c", else ""

The `relationships` dict parameter is the output of `get_all_relationships_for_file()` — keyed by chunk_id with "outgoing" and "incoming" lists.
</action>

<action>
Add a new `query` command to `02-worktrees/glma/src/glma/cli.py`:

```python
@app.command()
def query(
    filepath: str = typer.Argument(..., help="Path to file to query (relative to repo root)."),
    verbose: bool = typer.Option(False, "--verbose", "-V", help="Include full code bodies."),
    output: Optional[str] = typer.Option(None, "--output", "-o", help="Output file path (default: stdout)."),
    repo_root: Optional[Path] = typer.Option(None, "--repo", "-r", help="Repo root directory (auto-detected)."),
) -> None:
```

Implementation logic:

1. **Resolve repo root:**
   - If `--repo` provided, use it
   - Otherwise, walk up from CWD looking for `.glma-index/` directory or `.glma.toml` file
   - If not found, print "Error: Not inside an indexed repository" to stderr and `raise typer.Exit(4)`

2. **Locate index database:**
   - Open `{repo_root}/.glma-index/db/index.lbug` (matching pipeline.py's db_path pattern)
   - If doesn't exist, print "Error: No index found. Run `glma index` first." to stderr and `raise typer.Exit(4)`

3. **Check file exists on disk:**
   - `disk_path = repo_root / filepath`
   - If not `disk_path.exists()`, print f"Error: File not found: {filepath}" to stderr and `raise typer.Exit(1)`

4. **Look up file in index:**
   - `store = LadybugStore(db_path)` — opens the DB
   - `file_record = store.get_file_record(filepath)`
   - If `file_record is None`, print f"Error: File not indexed: {filepath}" to stderr and `raise typer.Exit(2)`

5. **Stale index check:**
   - Import `file_content_hash` from `glma.index.pipeline`
   - `current_hash = file_content_hash(disk_path)`
   - If `current_hash != file_record.content_hash`, print "Warning: File has been modified since last index. Results may be stale." to stderr (warning, not error — still show output, but set a flag for exit code 3)

6. **Load data and format:**
   - `chunks = store.get_chunks_for_file(filepath)`
   - `relationships = store.get_all_relationships_for_file(filepath)`
   - `from glma.query.formatter import format_compact_output`
   - `output_text = format_compact_output(filepath, file_record, chunks, relationships, verbose=verbose)`

7. **Output:**
   - If `--output` provided, write to that file path
   - Otherwise, print to stdout via `console.print(output_text, highlight=False, soft_wrap=True)`

8. **Exit code:**
   - If stale: `raise typer.Exit(3)` after printing output
   - Otherwise: normal exit (0)

Import `sys` at the top for stderr access. Use `sys.stderr.write()` for error messages to keep stdout clean.

Note: The full flag set from CONTEXT.md D-13 (`--depth`, `--no-relationships`, `--format json`, `--rel-types`, `--summary-only`) will be added in Plan 03-03. This plan only implements `--verbose` and `--output`.
</action>
